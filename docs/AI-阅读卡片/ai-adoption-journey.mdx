# 从怀疑到接纳：Mitchell Hashimoto 的 AI 使用之旅

<!-- truncate -->

<Callout type="success" title="一句话先看懂">
AI 的价值，不在于“会聊天”，而在于能进入工作流、形成闭环、并且让你保留控制感。
</Callout>

## 一、背景：为什么这篇文章值得看

### 1.1 一篇“从不买账到能用”的真实记录
这篇文章不是 AI 宣传稿，也不是技术狂热宣言，而是一篇非常个人化、非常克制的使用经验总结。

> 作者 Mitchell Hashimoto 讲的不是“AI 有多神”，而是自己如何一步步从怀疑 AI、排斥 AI，到最终把它纳入日常工作流。

### 1.2 为什么很多人一开始用 AI 都会失望
<Callout type="warning" title="核心前提">
新工具通常都会经历：**低效 → 勉强可用 → 真正有价值**。
</Callout>

- 第一个阶段，你会觉得它拖慢你
- 第二个阶段，你开始知道它能做什么、不能做什么
- 第三个阶段，它才真正进入你的工作流，甚至改变你的习惯

作者强调，很多人对 AI 的失望，并不代表 AI 没价值，而是还停留在“错误使用方式”里。

### 1.3 传统聊天式 AI 为什么不够用
作者的第一个结论很直接：**不要指望纯聊天界面完成真正的开发工作。**

| 问题 | 聊天式 AI 的表现 |
|---|---|
| 需要改代码 | 往往要来回复制粘贴 |
| 需要运行验证 | 没有外部行为闭环 |
| 需要持续迭代 | 容易中断上下文 |
| 需要提高效率 | 经常比自己做还慢 |

<Callout type="danger" title="结论">
如果只是把 AI 当成“更会说话的搜索框”，很难得到稳定收益。
</Callout>

## 二、是什么：作者如何逐步真正用起来

### 2.1 整体脉络
这篇文章把 AI 采用过程分成几个阶段：

1. **Drop the Chatbot**：停止把聊天框当成主要工作界面
2. **Reproduce Your Own Work**：先自己手做，再让 AI 重做一遍
3. **End-of-Day Agents**：把代理任务放到一天快结束的时候执行
4. **Outsource the Slam Dunks**：把高成功率任务外包给 AI
5. **Engineer the Harness**：为 AI 工具设计更好的验证和约束
6. **Always Have an Agent Running**：尽量让后台总有一个 agent 在跑

这条路线非常有意思，因为它不是靠“相信 AI”，而是靠**逐步建立可控性**。

### 2.2 各阶段分别在解决什么问题

#### 2.2.1 先放弃聊天框，转向 agent
聊天式界面对编码帮助有限。真正能产生价值的是 **agent**，也就是能读文件、执行程序、发 HTTP 请求的系统。

- 从“问答式使用”转向“执行式使用”
- 从“建议工具”转向“工作工具”
- 从“纯文本输出”转向“可行动闭环”

#### 2.2.2 先亲自重做，建立判断力
作者一开始对 Claude Code 也不满意，于是做了一件很费劲但很有效的事：**把自己手工做过的任务，再让 agent 重做一遍。**

<Callout type="tip" title="这一步的价值不在省时间">
它的真正价值，是帮你形成判断力：哪些任务 AI 擅长，哪些任务它不擅长，以及什么情况下根本不该用。
</Callout>

#### 2.2.3 把 agent 放到一天的尾声
作者后来发现，把 agent 任务放到工作日最后 30 分钟非常有效。原因不是 agent 突然变聪明了，而是这类时间本来就很难高效做深度工作。

这带来两个好处：

- 把低效时段转化为“后台产出”
- 第二天早上能直接接着处理结果，形成“warm start”

#### 2.2.4 把“容易成功”的任务外包出去
当作者对 AI 的能力边界有了更明确判断后，就开始把那些“AI 大概率能做好”的任务交给它，自己则继续做更需要人类判断的工作。

这一步很关键，因为它不是“什么都交给 AI”，而是：

- 选择成功率高的任务
- 把 agent 放在它最能发挥的地方
- 人类保留对复杂任务和整体方向的控制权

### 2.3 Harness engineering：这篇文章最有启发的部分
<Callout type="success" title="最重要的一句">
一旦发现 agent 会犯某类错误，就想办法设计系统，让它以后不要再犯。
</Callout>

这通常有两种做法：

1. **改 AGENTS.md 之类的隐式提示**
2. **加真正的程序化工具**，例如截图脚本、过滤测试脚本等

这说明 AI 的效率不是单靠模型，而是靠**工具、规则和反馈系统一起构成**。

## 三、一些有趣的点

### 3.1 “先自己做一遍，再让 AI 重做一遍”很笨，但很有效
这是文章里最反直觉的一点。它很耗时，但能让人真正知道：

- AI 到底会什么
- 它错在哪里
- 什么程度的输出才算可用
- 自己到底该在哪些场景中信任它

> 这其实是一种很强的学习方式：不是看别人总结，而是自己亲手建立认知。

### 3.2 这篇文章的核心不是效率，而是控制感
作者不断强调一个事实：**AI 真正有价值，不是因为它能替代你，而是因为它能在你掌控下帮你干活。**

| 维度 | 低价值使用 | 高价值使用 |
|---|---|---|
| 使用方式 | 聊天 | agent |
| 工作模式 | 被动等待 | 主动调度 |
| 结果质量 | 反复返工 | 可验证产出 |
| 人的角色 | 纠错机器 | 任务设计者 |

### 3.3 对“技能形成”的担忧也很现实
作者承认，长期依赖 AI 可能会影响技能形成，尤其是对基础不扎实的人。

<Callout type="warning" title="提醒">
AI 不是免费送你的能力；它可能替你完成任务，但不一定替你长本事。
</Callout>

### 3.4 作者并不把 AI 神化
文章最好的地方，是它没有极端化。作者既承认 AI 带来效率，也承认它的局限；既承认自己会持续使用，也承认别人完全可以出于自己的理由选择不用。

这让整篇文章的气质很稳，不像广告，更像一个成熟工程师的阶段性记录。

## 四、总结与反思

### 4.1 总结全文
这篇文章讲的是一个很务实的过程：作者从不信 AI，到把聊天式工具排除出主流程，再通过“自己做一遍、让 agent 重做一遍”的方式建立判断力，最后逐步把 AI 纳入日常工作流，并通过 harness engineering 提升 agent 的可用性。

### 4.2 能用在开发中的哪些点
1. **不要把聊天框当成主要工作界面**。真正有价值的是能执行、能验证的 agent。
2. **先理解任务边界，再决定是否交给 AI**。不是所有工作都该自动化。
3. **让 AI 做它擅长的事**。尤其是高成功率、可验证、重复性强的任务。
4. **把工具链做成闭环**。一旦发现错误，就改提示、加脚本、补验证。
5. **用 AI 的同时保留自己的判断力**。效率和能力成长要同时考虑。

### 4.3 对未来的启发
这篇文章真正给出的启发不是“AI 有多强”，而是：

- AI 工具的价值来自 workflow，而不是 demo
- 人类和 agent 的协作方式，会越来越像工程分工
- 未来最重要的不是更会聊天的模型，而是更可靠的执行系统

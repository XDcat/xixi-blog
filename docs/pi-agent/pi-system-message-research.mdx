# 从源码看 pi agent 的 system message 是怎么拼出来的

> 本文梳理 `coding-agent` 相关源码中 system message 的收集、重建与拼接顺序，便于排查「某段话从哪来、为何出现或消失」。

## 一、先说结论

pi agent 的 system message 不是一段固定字符串，而是一条分层构建链：最上层是资源与配置输入，中间是 session 的重建逻辑，底层才是字符串拼接与 provider 字段映射。

要找的不是「某一个 system prompt 文件在哪」，而是「它如何被一步步拼出来」。

## 二、system message 的来源有哪些

先把来源说清楚，再谈拼接顺序。

### 2.1 显式传入 / CLI 覆盖
`packages/coding-agent/src/main.ts` 会把命令行参数传给 `DefaultResourceLoader`，包括：
- `systemPrompt`
- `appendSystemPrompt`
- 额外 skills / prompts / themes / extensions

`DefaultResourceLoader` 还支持 override 回调，因此上层可以直接替换加载结果。

### 2.2 本地文件
`packages/coding-agent/src/core/resource-loader.ts` 会发现并读取这些文件：
- 项目 `.pi/` 目录下的 `SYSTEM.md`
- 项目 `.pi/` 目录下的 `APPEND_SYSTEM.md`
- agent 目录下的 `SYSTEM.md`
- agent 目录下的 `APPEND_SYSTEM.md`

:::caution[路径约定]
源码里通过 `join(this.cwd, CONFIG_DIR_NAME, "SYSTEM.md")` 查找文件，因此项目级 system prompt 落在 `.pi/` 下，而不是仓库根目录。
:::

此外，它还会向上遍历当前工作目录祖先目录，加载：
- `AGENTS.md`
- `CLAUDE.md`

这些内容会作为 project context 进入最终 prompt。

### 2.3 技能、模板、主题、扩展
资源加载器还会加载：
- skills
- prompt templates
- themes
- extensions

其中 skills 会被格式化成 `<available_skills>` 区块，直接写进 system prompt。

### 2.4 内置字符串
真正的 prompt 骨架来自 `packages/coding-agent/src/core/system-prompt.ts`，包括：
- 固定身份句子
- tools / guidelines
- Pi 文档引用块
- project context 块
- skills 块
- 当前日期和 cwd

## 三、自顶向下的构建链路

这一部分只回答：材料怎么进来、怎么被选中、怎么被拼起来。

### 3.1 第一步：启动时加载资源
入口在 `packages/coding-agent/src/main.ts`。

关键代码（真实源码片段）：

```ts
// main.ts
const resourceLoader = new DefaultResourceLoader({
  cwd,
  agentDir,
  settingsManager,
  additionalExtensionPaths: firstPass.extensions,
  additionalSkillPaths: firstPass.skills,
  additionalPromptTemplatePaths: firstPass.promptTemplates,
  additionalThemePaths: firstPass.themes,
  noExtensions: firstPass.noExtensions,
  noSkills: firstPass.noSkills,
  noPromptTemplates: firstPass.noPromptTemplates,
  noThemes: firstPass.noThemes,
  systemPrompt: firstPass.systemPrompt,
  appendSystemPrompt: firstPass.appendSystemPrompt,
});

await resourceLoader.reload();
```

这里可以看到：
- CLI 参数直接传入 `DefaultResourceLoader`
- `reload()` 是所有资源扫描的起点

这一层负责把外部材料都准备好。

### 3.2 第二步：资源加载器收集所有输入
`packages/coding-agent/src/core/resource-loader.ts` 负责收集并整理所有输入。
核心逻辑片段（真实源码结构）：

```ts
// resource-loader.ts (reload 末尾关键逻辑)
const baseSystemPrompt = resolvePromptInput(
  this.systemPromptSource ?? this.discoverSystemPromptFile(),
  "system prompt",
);
this.systemPrompt = this.systemPromptOverride
  ? this.systemPromptOverride(baseSystemPrompt)
  : baseSystemPrompt;

const appendSource =
  this.appendSystemPromptSource ?? this.discoverAppendSystemPromptFile();
const resolvedAppend = resolvePromptInput(
  appendSource,
  "append system prompt",
);
const baseAppend = resolvedAppend ? [resolvedAppend] : [];
this.appendSystemPrompt = this.appendSystemPromptOverride
  ? this.appendSystemPromptOverride(baseAppend)
  : baseAppend;
```

其中 `discoverSystemPromptFile()` 实现为：

```ts
private discoverSystemPromptFile(): string | undefined {
  const projectPath = join(this.cwd, CONFIG_DIR_NAME, "SYSTEM.md");
  if (existsSync(projectPath)) return projectPath;

  const globalPath = join(this.agentDir, "SYSTEM.md");
  if (existsSync(globalPath)) return globalPath;

  return undefined;
}
```

可以看到真实优先级为：
1. CLI 传入的 `--system-prompt`
2. 项目 `.pi/SYSTEM.md`
3. agentDir 下的 `SYSTEM.md`
4. 最终还可以被 `systemPromptOverride` 钩子修改

:::info[append 在 loader 与 session 中的形态]
- `appendSystemPrompt` 在 loader 内部是 `string[]`（单个来源也会包成数组）
- `join("\n\n")` 发生在 `AgentSession`，不在 resource loader
:::

它本身不拼 prompt，只负责整理材料：
- 找 `SYSTEM.md` / `APPEND_SYSTEM.md`
- 找 `AGENTS.md` / `CLAUDE.md`
- 加载 skills、prompts、themes
- 收集扩展资源
- 生成 `agentsFiles`

### 3.3 第三步：AgentSession 决定当前轮用什么材料
`packages/coding-agent/src/core/agent-session.ts` 里的 `_rebuildSystemPrompt()` 会把相关材料重新汇总。
核心代码（真实源码片段）：

```ts
// agent-session.ts
private _rebuildSystemPrompt(toolNames: string[]): string {
  const validToolNames = toolNames.filter((name) =>
    this._toolRegistry.has(name),
  );

  const toolSnippets: Record<string, string> = {};
  const promptGuidelines: string[] = [];

  for (const name of validToolNames) {
    const snippet = this._toolPromptSnippets.get(name);
    if (snippet) {
      toolSnippets[name] = snippet;
    }

    const toolGuidelines = this._toolPromptGuidelines.get(name);
    if (toolGuidelines) {
      promptGuidelines.push(...toolGuidelines);
    }
  }

  const loaderSystemPrompt = this._resourceLoader.getSystemPrompt();
  const loaderAppendSystemPrompt =
    this._resourceLoader.getAppendSystemPrompt();
  const appendSystemPrompt =
    loaderAppendSystemPrompt.length > 0
      ? loaderAppendSystemPrompt.join("\n\n")
      : undefined;

  const loadedSkills = this._resourceLoader.getSkills().skills;
  const loadedContextFiles =
    this._resourceLoader.getAgentsFiles().agentsFiles;

  return buildSystemPrompt({
    cwd: this._cwd,
    skills: loadedSkills,
    contextFiles: loadedContextFiles,
    customPrompt: loaderSystemPrompt,
    appendSystemPrompt,
    selectedTools: validToolNames,
    toolSnippets,
    promptGuidelines,
  });
}
```

`appendSystemPrompt` 在此才被 `join("\n\n")`，与 loader 里保持数组形态相对应。工具不仅影响能否调用，也决定写入 system prompt 的 snippet 与 guideline；工具集变化（例如 `setActiveToolsByName()`）会触发 base system prompt 重建。

`_rebuildSystemPrompt()` 会把：
- loader 的 system prompt
- loader 的 append prompt
- skills
- agentsFiles / project context
- 当前激活工具的 snippets 和 guidelines

统一交给 `buildSystemPrompt()`。工具集变化会触发 prompt 重建，因此 **tools 的说明也是 system message 的一部分**。

### 3.4 第四步：真正拼接字符串
`packages/coding-agent/src/core/system-prompt.ts` 的 `buildSystemPrompt()` 负责最终拼装。
关键代码结构（真实源码片段，保留核心分支）：

```ts
export function buildSystemPrompt(options: BuildSystemPromptOptions = {}): string {
  const {
    customPrompt,
    selectedTools,
    toolSnippets,
    promptGuidelines,
    appendSystemPrompt,
    cwd,
    contextFiles: providedContextFiles,
    skills: providedSkills,
  } = options;

  const date = new Date().toISOString().slice(0, 10);
  const resolvedCwd = cwd ?? process.cwd();

  // ✅ 分支一：存在 customPrompt（来自 SYSTEM.md 或 CLI）
  if (customPrompt) {
    let prompt = customPrompt;

    if (appendSystemPrompt) {
      prompt += `\n\n${appendSystemPrompt}`;
    }

    if (providedContextFiles?.length) {
      prompt += "\n\n# Project Context\n\n";
      for (const { path, content } of providedContextFiles) {
        prompt += `## ${path}\n\n${content}\n\n`;
      }
    }

    const hasRead = !selectedTools || selectedTools.includes("read");
    if (hasRead && providedSkills?.length) {
      prompt += formatSkillsForPrompt(providedSkills);
    }

    prompt += `\nCurrent date: ${date}`;
    prompt += `\nCurrent working directory: ${resolvedCwd}`;

    return prompt;
  }

  // ✅ 分支二：使用内置默认模板（部分省略）
  const tools = selectedTools || ["read", "bash", "edit", "write"];
  const visibleTools = tools.filter((name) => !!toolSnippets?.[name]);

  // 只有有 snippet 的工具才会出现在 Available tools 中
  const toolsList =
    visibleTools.length > 0
      ? visibleTools.map((name) => `- ${name}: ${toolSnippets![name]}`).join("\n")
      : "(none)";

  // 根据可用工具动态生成 guidelines（部分省略）

  let prompt = `You are an expert coding assistant operating inside pi...`;

  if (appendSystemPrompt) {
    prompt += `\n\n${appendSystemPrompt}`;
  }

  if (providedContextFiles?.length) {
    prompt += "\n\n# Project Context\n\n";
    for (const { path, content } of providedContextFiles) {
      prompt += `## ${path}\n\n${content}\n\n`;
    }
  }

  const hasRead = tools.includes("read");
  if (hasRead && providedSkills?.length) {
    prompt += formatSkillsForPrompt(providedSkills);
  }

  prompt += `\nCurrent date: ${date}`;
  prompt += `\nCurrent working directory: ${resolvedCwd}`;

  return prompt;
}
```

关键结论：

- 存在 `customPrompt`（来自 `SYSTEM.md` 或 CLI）时，不会走内置默认模板，但仍会追加 append、Project Context、Skills（在条件满足时）、日期与 cwd
- Skills 仅在 `read` 工具启用时才会写入 prompt
- 工具要有 snippet 才会出现在「Available tools」列表中；guidelines 随工具集动态生成，并会追加固定规则

### 3.5 第五步：送进 provider

最终 `context.systemPrompt` 会进入 `packages/ai/src/providers/*`。例如 OpenAI provider（简化）：

```ts
// openai.ts
const response = await client.responses.create({
  model,
  instructions: context.systemPrompt,
  input: messages,
})
```

Anthropic：

```ts
// anthropic.ts
client.messages.create({
  system: context.systemPrompt,
  messages,
})
```

可以看到 provider 层只做字段映射，不修改语义。

不同 provider 对同一份 prompt 的落点不同：
- Google：`systemInstruction`
- Anthropic：`params.system`
- Mistral：`messages` 里的 `system` role
- OpenAI Responses / Codex Responses：`instructions`

所以「system message」在内部是统一概念，落在外部 API 上却是不同字段。

## 四、本地文件、优先级与排查顺序

### 4.1 参与构造的文件

便于对照排查时，可按下面清单核对（含非「纯 system」但会进上下文的项）：

- `SYSTEM.md`、`APPEND_SYSTEM.md`（主用与追加）
- `AGENTS.md`、`CLAUDE.md`（多层级叠加进 Project Context，不替代 `SYSTEM.md`）
- skills 下的 `SKILL.md`、prompt templates、extensions 提供的资源
- 运行时注入的当前日期与工作目录

### 4.2 覆盖顺序（由谁说了算）

粗粒度优先级可以理解成：

1. 显式传入 / CLI / override 回调
2. 项目本地（`.pi/`）文件
3. agent 目录下的全局文件
4. 无自定义 `SYSTEM.md` 时的 `buildSystemPrompt()` 内置模板
5. 扩展在单轮对 prompt 的修改
6. provider 适配层把同一串文本映射到各家 API 字段

输入层优先于默认模板；模板层再高于「怎么塞进 HTTP 请求」的适配层。

#### 4.2.1 源码里谁先谁后

- `resource-loader.ts`：决定哪些文件被读入、override 如何生效
- `agent-session.ts`：决定当前轮哪些资源进入 `buildSystemPrompt`
- `system-prompt.ts`：按固定顺序拼接字符串
- `packages/ai/src/providers/*`：只映射字段，不改语义

排查「某段为什么会出现/没出现」时，应先看 loader 与 session，而不是 provider。

## 五、几个容易忽略的细节

### 5.1 细节清单

有些行为不写在主流程里，却会改变最终 prompt。

##### a. skills 不是全量展示

`formatSkillsForPrompt()` 会过滤掉 `disableModelInvocation=true` 的 skill。

对应源码（简化）：

```ts
export function formatSkillsForPrompt(skills: Skill[]) {
  return skills
    .filter(s => !s.disableModelInvocation)
    .map(s => renderSkill(s))
    .join("\n")
}
```

这类 skill 不会出现在模型可见的技能列表里，只能通过显式命令调用。

##### b. system prompt 会因工具集变化而重建

`setActiveToolsByName()` 会重新构建 base system prompt，因此工具说明与 system prompt 绑定，而不是彼此独立。

##### c. 扩展多作「本轮修改」

扩展 runner 可以在 turn 级别改 system prompt，但 session 会在必要时恢复 base prompt，故扩展更像临时注入，而非持久改写。

##### d. `SYSTEM.md` / `APPEND_SYSTEM.md` 并非唯一输入形态

`resolvePromptInput()` 既接受路径也接受直接文本，因此 CLI 等外部传参的优先级仍然很高。

## 六、结构组成与实践建议

落地使用时，先看 **system message 由哪些块组成、每一块从哪来**；再谈怎么拆文件、放内容。

本节顺序与 `buildSystemPrompt()` 中的拼接一致。

### 6.1 system message 的结构组成（按拼接顺序）
根据 `buildSystemPrompt()` 的真实实现，最终的 system message 由以下几部分组成：

1. **基础模板或自定义 SYSTEM.md（必有其一）**
   - 来源：
     - CLI `--system-prompt`
     - 项目 `.pi/SYSTEM.md`
     - agentDir 下的 `SYSTEM.md`
   - 规则：三选一（按优先级覆盖），不会叠加多个 SYSTEM.md。

2. **APPEND_SYSTEM.md（可选）**
   - 来源：
     - CLI `--append-system-prompt`
     - 项目 `.pi/APPEND_SYSTEM.md`
     - agentDir 下的 `APPEND_SYSTEM.md`
   - 规则：只会选择一个来源，但在 AgentSession 中会被追加到主 prompt 之后。

3. **Project Context（AGENTS.md / CLAUDE.md）**
   - 来源：
     - agentDir 下的 `AGENTS.md` / `CLAUDE.md`
     - 当前工作目录及其所有祖先目录中的 `AGENTS.md` / `CLAUDE.md`
   - 规则：
     - 可能加载多个文件
     - 会全部拼接进 `# Project Context` 区块
     - 不互相覆盖，而是按目录层级顺序叠加

4. **Skills 区块**
   - 来源：
     - resource-loader 加载的所有启用 skill
     - 具体文件通常是 `SKILL.md`
   - 规则：
     - 会加载多个 skill
     - 过滤 `disableModelInvocation=true`
     - 只有在 `read` 工具启用时才会加入 system prompt

5. **工具说明（Available tools + Guidelines）**
   - 来源：
     - 当前激活的工具集合
     - 每个工具的 snippet 与 guideline
   - 规则：
     - 只有存在 snippet 的工具才会出现在“Available tools”列表
     - guideline 会根据工具动态生成
     - 始终会追加固定 guideline（如“Be concise…”）

6. **运行时信息**
   - 当前日期
   - 当前工作目录
   - 始终追加在最后

---

### 6.2 哪些是“选择一个”，哪些是“全部叠加”？
| 部分 | 来源数量 | 叠加方式 |
|------|----------|----------|
| SYSTEM.md | 多选一 | 按优先级覆盖 |
| APPEND_SYSTEM.md | 多选一 | 追加一次 |
| AGENTS.md / CLAUDE.md | 可多个 | 全部叠加 |
| Skills | 可多个 | 全部叠加（有过滤） |
| Tool snippets | 可多个 | 按 active tools 选择 |
| 固定模板 | 仅在无 SYSTEM.md 时启用 | 单一模板 |

理解这一点非常重要：

> SYSTEM.md 是“替换底稿”，而 AGENTS.md / Skills 是“附加区块”。

---

### 6.3 基于源码结构的实践建议

搞清楚构造顺序后，放内容时可以按下面方式分工。

#### 6.3.1 身份与长期规则 → `SYSTEM.md`

路径示例：`项目/.pi/SYSTEM.md`。

适合：角色设定、代码风格、长期团队规范。它会作为整段 prompt 的底稿。

#### 6.3.2 补充规则 → `APPEND_SYSTEM.md`

适合：临时强化、阶段说明、不想动主 `SYSTEM.md` 时的附加约束；总是接在主线文案之后。

#### 6.3.3 项目背景 → `AGENTS.md` / `CLAUDE.md`

会向上遍历、多文件叠加，落在 `# Project Context`。适合架构、依赖、构建方式等；**不**替代 `SYSTEM.md`。

#### 6.3.4 可复用流程 → Skill

结构化、可被显式调用、按需启用；避免把长流程全堆进 `SYSTEM.md`。

#### 6.3.5 控制工具 snippet 体量

snippet 决定工具是否在列表中「可见」；过长会膨胀 prompt，且 guideline 仍随工具生成。建议短、只暴露必要工具。

#### 6.3.6 扩展改 prompt 时分清作用域

扩展可在单轮调用：

```ts
agent.setSystemPrompt(result.systemPrompt)
```

这只影响当轮呈现，一般不会等同修改 `_baseSystemPrompt`；**长期规则仍应写进配置文件，而不是依赖扩展临时注入。**

## 七、总结

:::tip[一条结构口诀]
底稿 + 追加块 + 上下文块 + 技能块 + 工具块 + 运行时信息，按顺序线性拼接。
:::

**整条链路**：资源加载器收集材料 → `AgentSession` 决定当前轮参与构建的内容 → `buildSystemPrompt` 拼成字符串 → provider adapter 映射到各厂商 API 字段。

这样分层的好处是职责清晰、易 override、易加扩展，又能消化不同 API 对 system 字段的差异。

若只记一句排查顺序：**先追内容从哪来（loader / session），再看怎么拼（`system-prompt`），最后才看怎么发（provider）。**

# 「分析」System Prompt

<!-- truncate -->

<ReadingCardMeta tags={["OpenClaw", "System Prompt", "Agent", "Memory"]} source="https://www.yuque.com/xiaodai-tehze/lah8hg/bzhocz1ec68i4vg0" dateRead="2026-03-29" />

## 一、总体印象

这份 System Prompt 不是普通的聊天角色设定，而是 OpenClaw 整个 Agent 运行机制的说明书。它定义了：

- 工具怎么用
- Skills 怎么发现和执行
- 记忆怎么读写
- 会话怎么压缩
- 心跳和 cron 怎么工作
- 如何在不同 channel 里回复

换句话说，它不是只在回答问题，而是在规定一个 Agent 的工作方式。

## 二、工具体系：这不是聊天框，是可执行系统

### 2.1 工具列表
工具被严格区分并按策略过滤。核心工具包括：

- `read`：读取文件
- `write`：创建或覆盖文件
- `edit`：精确修改文件
- `exec`：执行 Shell 命令
- `process`：管理后台进程
- `web_search` / `web_fetch`：搜索与抓取网页
- `browser` / `canvas`：浏览器与画布操作
- `nodes`：配对节点设备
- `cron`：定时任务
- `message`：消息与频道操作
- `gateway`：重启/更新/配置 OpenClaw 进程
- `sessions_*`：跨会话协作
- `memory_*`：长期记忆搜索与读取
- `tts`：语音输出

这套工具体系说明，系统不是让模型“说得更像人”，而是让它“真的能做事”。

### 2.2 工具调用风格
System Prompt 对工具调用风格要求很清楚：

- 常规、低风险的工具调用可以直接执行，不必铺垫
- 多步骤、复杂、敏感操作才需要简短说明
- 有一级工具就优先用一级工具，而不是让用户自己跑 CLI
- 如果 exec 返回审批待定，要把完整命令原样展示给用户审批

这说明它追求的是可执行、可审计、可控。

## 三、Skills：把“技能”当成 Agent 的能力层

### 3.1 Skills 的定位
System Prompt 强制要求在回复前扫描 available_skills 的描述，判断是否有明确适用的 skill。只有当 skill 明确适用时，才读取对应的 `SKILL.md`。

这有几个重要信号：

- Skills 是能力，而不是装饰
- Skills 有明确边界，不是无限膨胀的工具箱
- 模型不应该乱读技能，也不应该到处乱用

### 3.2 为什么 Skills 重要
它让 Agent 从“会说话”变成“会按场景调用专门能力”。

比如：

- 管理 Apple 提醒事项
- 处理 GitHub issues
- 做视频帧提取
- 处理天气
- 管理 Things 3

这让 OpenClaw 更像一个“带职业技能的助理”，而不是纯聊天机器人。

## 四、记忆：长期记忆不是摆设，而是 Agent 的人格连续性

### 4.1 Memory Recall
这份 Prompt 明确要求：在回答关于过往工作、决策、日期、人物、偏好或待办事项的问题前，必须先进行 memory_search。

这意味着：

- 记忆不是可选项
- 回答历史相关问题时，不能凭印象胡说
- 记忆是一种外置状态，而不是模型自己脑补

### 4.2 Memory 的工作方式
- `MEMORY.md`：长期记忆，主会话可读写
- `memory/YYYY-MM-DD.md`：每日原始日志
- 要记住东西，就写文件，不要只“脑子里记着”

这其实是一种非常实用的 Agent 连续性设计。

## 五、上下文压缩：让长会话不会被自己拖死

### 5.1 Tool 上下文压缩
工具结果可以被压缩隐藏，但请求参数仍保留。这说明系统关心的是：

- 保留关键决策链
- 减少无意义的历史膨胀
- 让 Agent 继续工作，而不是被历史淹没

### 5.2 会话压缩
当历史会话太长时，系统会把前面的内容压成摘要，再放进新的 System Prompt 下面。

这意味着长任务可以持续推进，而不必一直携带全部原始上下文。

OpenClaw 对长任务的思路不是无限记忆，而是摘要化状态保存。

## 六、心跳与 cron：让系统自己动起来

### 6.1 心跳
心跳用于周期性检查 `HEARTBEAT.md`，适合做一些可延迟的批量检查：

- 邮件
- 日历
- 提及
- 天气

### 6.2 cron
cron 适合精确时间的任务，比如提醒和定时执行。

这两者区分得很清楚：

- 心跳：更像顺手检查一下
- cron：更像精准在某个时刻触发

## 七、如何在执行过程中插话

OpenClaw 使用 FIFO 队列处理消息，并支持不同模式：

- `steer`：中途快速插话，改变方向
- `followup`：等当前任务自然跑完
- `collect`：等任务跑完后再合并多条消息

这套机制的价值是：

- 让并发消息有秩序
- 让用户既可以打断，也可以等待
- 让 Agent 的执行流更可控

## 八、我的理解：这份 Prompt 的真正价值是什么？

### 8.1 它不是提示词，而是运行规范
这份 System Prompt 的本质，不只是告诉模型“你是谁”，而是在规定：

- 你能做什么
- 怎么做
- 何时做
- 何时该停
- 如何继续记住自己

### 8.2 它把 OpenClaw 的核心能力统一起来了
可以把这份 Prompt 理解成一套完整的 Agent OS 说明：

- Channel / Gateway / Env
- ReAct 自驱动循环
- Skills 体系
- 长短期记忆
- 上下文压缩
- 心跳 / cron
- 跨会话协作

结论很直接：OpenClaw 的 System Prompt 不是在训练一个更会聊天的助手，而是在定义一个可持续运行的 Agent 系统。

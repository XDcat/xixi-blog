# 🦞OpenClaw 原理解析与思考总结

<Callout type="info" title="一句话总结">
OpenClaw 本质上是一个“全副武装的 Agent”：把对话、工具、技能、记忆、上下文压缩和 24h 运行环境整合到一起。
</Callout>

## 一、OpenClaw 到底是什么？

### 1.1 OpenClaw 是什么？
<Callout type="warning" title="核心判断">
OpenClaw 本质上是一个「Agent」，或者说是一个「全副武装的 Agent」。
</Callout>

OpenClaw 是一个产品上的创新，让「大众」能够「真正便捷」地用到 AI 的众多生态，例如 Agent、MCP、Skills 等。技术上的创新，是对 AI 生态进行整合；交互上的创新，是提出了「24h 不停机」的理念。底层技术原理仍然是现有那一套，例如 Agent 自主规划执行、Skills 等。

### 1.2 为什么说 OpenClaw 是做 AI 整合，而不是底层架构升级？
近年来 AI 工具从单纯的 Chat，发展到 Agent、MCP、Skills 等丰富生态。OpenClaw 真正做的事情，是提高这些工具使用的下限和效果，而不是改变它们能力的上限。

| AI 工具 | 代表 | 特点 | 关键技术 |
| --- | --- | --- | --- |
| Chat 对话 | ChatGPT | 只能对话，切换会话后记忆清空 | Transformer |
| 编码工具 | Cursor | 代码 Tab 续写；Agent 做局部代码新增与修改 | Code Tab, Agent |
| 个人助理 | Manus | 与其他软件交互，自动化做编码以外的事情 | ReAct |
| 增强版 AI 编码工具 | Claude Code | 更加自动化；借助 MCP、Skills、记忆能力完成更复杂任务 | MCP, Skills, ACP |

### 1.3 OpenClaw 解决的问题？
![OpenClaw 解决的问题](https://cdn.nlark.com/yuque/0/2026/png/43125484/1773631698366-ccf458b3-b169-489a-9cdf-9a575dda651f.png)

它解决两个问题：

1. **对非程序员来说**，让人真正用上 Skills、Memory、MCP 等新能力，而不是只停留在 Chat。
2. **对程序员来说**，让 AI 能 24 小时不停地干活。

#### 从非程序员角度看
普通人真正接触到的往往还是 Chat 对话工具；对长期记忆、MCP、Skills 这类能力并不熟悉。OpenClaw 的意义，是把这些能力变成“能被日常使用的东西”。

普通人的核心需求大概有两类：

- 能自定义机器人，并记住之前做过的事情 → 长期记忆
- 能真正做事情，例如管理邮件、分析股票、写文档 → MCP、Skills

而这些，OpenClaw 做到了。

#### 从程序员角度看
增强版 AI 编码工具已经能覆盖很多能力，但使用方式仍有问题：

- 需要在特定电脑上、用特定方式对话，例如 CLI 或 IDE
- 一旦关掉电脑，任务就停了
- 仍然更偏编码任务，难以和日常工具打通

<Callout type="danger" title="关键点">
即使有了增强版 AI 编码工具，仍然常常是“程序员陪着 AI 做事情”，而 OpenClaw 试图把任务变成 24h 常驻运行。
</Callout>

### 1.4 为什么 OpenClaw 能够火起来？
原因很直接：

- 对用户来说，它解决了“AI 工具不好用、用不上”的问题
- 对 AI 公司来说，它把 token 的消耗场景做大了
- 对技术发展来说，它把零散能力整合成了一个现象级产品
- 现实驱动也很强：落后就要挨打，必须推，也不得不推

### 1.5 OpenClaw 对程序员的影响是什么？
作为程序员，更合适的态度不是恐慌，而是持续学习。

- **不要恐慌**：AI 对程序员的冲击不是突然发生的，未来也只会更多。没必要每次技术更新都紧张一次。
- **持续学习**：OpenClaw 已经证明自己是一款成功产品，还是开源的，更值得去学习它、理解它、吸收它。

## 三、关键技术点

### 3.1 基本框架：Channel + Gateway + Env
OpenClaw 的关键点在于两件事：

1. 提供完善的本地交互环境
2. 通过 Gateway 接入不同的 App 进行 Chat

可以理解成以下四层：

- **Channel 层**：不同的聊天方式
- **Gateway 层**：会话管理与路由
- **执行层**：本地环境 + Agent 执行
- **拓展层**：大模型提供商 + 沙箱环境

### 3.2 智能体循环 Agent Loop：ReAct 自我驱动的 Agent
官方把 Agent 执行的循环叫做 Agent Loop，但我更愿意把它理解成 ReAct 范式。

#### ReAct 模式的 Agent
核心点是实现了“规划—执行—反思”的循环。驱动类型有两种：

- 等待用户对话后继续
- 执行 Tool 后自动继续

对于复杂任务，大量循环是由 Tool 自动驱动的，不需要用户频繁介入。

#### 底层具体实现：Pi Agent
Pi Agent 的定位与 Claude Code 类似，是一款增强版 AI 编码工具。它的核心思想是尽可能简单，主要依赖 read/write/edit/bash 四个 tool，追求可预测性。

| 对比维度 | Claude Code | Codex | Pi |
| :--- | :--- | :--- | :--- |
| 系统提示词 | 约 10,000+ tokens | 适中 | < 1,000 tokens |
| 内置工具数 | 数十个 | 适中 | 4 个 |
| Plan Mode | 有 | 有 | 无（用文件代替） |
| MCP 支持 | 有 | 有 | 无（用 CLI 工具代替） |
| Sub-Agent | 有（不可观测） | — | 无（通过 bash 自我调用） |

如果想了解更多，见：[下一代Agent架构——Pi Agent Core 设计逻辑深度解析 - 知乎](https://zhuanlan.zhihu.com/p/2004665077618458930)

### 3.3 自主进化：沉淀长期记忆
长期记忆以 markdown 的形式存储在 Agent 的 workspace 中，例如 `~/.openclaw/workspace/`。

读取长期记忆会自动注入上下文，包括：

- 动态注入到 System Prompt
- 通过 `read`、`memory_search` 等 tool 注入

更新则包括：

- 用户主动说“记住这个”触发
- 新建会话、定时任务等自动更新

### 3.4 上下文压缩
#### 3.4.1 Tool 上下文压缩
隐藏历史 tool 的结果，但请求参数不会被隐藏。

#### 3.4.2 会话压缩
使用摘要压缩历史对话记录，可以减少大量消息。压缩后，System Prompt 下面直接接摘要，而不是完整历史会话。

### 3.5 自动任务执行：心跳与定时任务
OpenClaw 中有两种自动任务执行策略：

- **心跳**：每隔固定时间检查 HEARTBEAT.md 中指定的任务
- **cron 定时任务**：精准在某个时刻执行任务

这个机制让 OpenClaw 更主动地做事情。

### 3.6 如何在 OpenClaw 执行过程中插话？
OpenClaw 使用 FIFO 队列来处理可能并发的消息。简化来说：

- `steer`：中途快速插话，改变方向
- `followup`：任务跑完后再处理
- `collect`：任务跑完后合并多条消息再处理

**总结：**
- steer 系列适合中途改方向
- followup / collect 更保守，适合不打断原流程

## 四、最佳实践

- 在使用 OpenClaw 前，先想想自己是否真的需要它
- 准备独立机器，24h 运行，随时接受指令
- Skills 是提升 AI 能力的关键
- 多用多体验，但不要盲目信任
- Token 太贵，尽量用公司提供的 API
- 多看看官方文档

## 五、总结与反思

### 5.1 未来需要更多面向 AI 的应用，为 AI 提供更好的交互环境
我现在写文档会用语雀等在线文档，但无法让 AI 帮我润色。我期待这类工具能提供 API，让 AI 像改本地文件一样修改文档。

### 5.2 如何将 OpenClaw 应用到实际项目中？
对于个人或团队，OpenClaw 有助于编码等日常任务效率提升。但如果放到真实 toB / toC 应用中，很多优势会被削弱。

比如：

- 不能为每个用户提供独立环境
- 共用环境是否撑得住复杂任务
- 客服机器人场景虽然可行，但不一定需要本地环境执行
- 定时任务和心跳机制在很多业务里意义不大

<Callout type="success" title="我的结论">
OpenClaw 大多数优势都被剥离后，剩下的是 ReAct 自主规划、Tools/Skills 调用、长短期记忆管理、上下文压缩等通用 Agent 能力。对应用开发来说，不是要照搬“小龙虾壳”，而是学习其中的 Agent 设计思想。
</Callout>

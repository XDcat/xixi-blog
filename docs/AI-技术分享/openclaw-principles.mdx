# 🦞OpenClaw 原理解析与思考总结
## 一、OpenClaw 到底是什么？
### 1.1 OpenClaw 是什么？
:::tip
OpenClaw 本质上是一个「Agent」，或者说是一个「全副武装的Agent」。

:::

OpenClaw 是一个产品上的创新，让「大众」能够「真正便捷」的用到 AI 的众多生态，例如Agent、MCP、Skills 等。技术上的创新是「对 AI 生态进行整合」，以及在交互上提出了「24h 不停机」的理念。但是底层技术原理还是现有的那一套，如 Agent 自主规划执行、Skills 等。

### 1.2 为什么说 OpenClaw 是做 AI 整合，而不是底层架构升级？
下表是近年来 AI 工具的发展中标志性的工具，从最初单纯的 Chat 到如今丰富的生态（Agent、MCP、Skills 等）。而 OpenClaw 真正能做的事情，无法超出以下工具能力的范围。**<font style={{ backgroundColor: '#FBDE28' }}>通过后面章节了解其原理后，相信你也会有类似的想法。</font>**

:::tip
OpenClaw 真正做的是提高以下工具使用成本和效果的下限，但无法改变上限。

:::

| **AI 工具** | **代表** | **特点** | **关键技术** |
| --- | --- | --- | --- |
| Chat 对话 | ChatGPT | 只能对话，切换会话后记忆清空。 | Transformer |
| 编码工具 | Cursor | 代码 Tab 续写；Agent 去完成局部代码的新增后修改。 | Code Tab, Agent |
| 个人助理 | Manus | 与其他软件交互，自动化做编码以外的事情。 | ReAct |
| 增强版 AI 编码工具 | Claude Code | 更加「自动化」<br/>+ 感知能力：MCP 和外部软件进行交互<br/>+ 专业能力：Skills 做特定事情上更加稳定高效<br/>+ 记忆能力：让模型能够记住会话、偏好、经验等 | MCP, Skills, ACP |


### 1.3 OpenClaw 解决的问题？

![](/images/yuque/e356865f1ea2.png)

它解决两个问题：

1. **对于非程序员来说，能够真正用上 Skills、Memory、MCP 等较新的 AI 技术，而不仅仅停留在 Chat 模式。**
2. **对于程序员来说，能够让 AI 24 小时不停的干活。这也是 Openclaw 诞生的原因。**

---

**从非程序员的角度分析**

AI 应用技术发展的很快，但对于普通人真正能用到多少呢？以上一节的表格为例，真正用到的还是 Chat 对话工具，例如豆包、千问、deepseek等；普通人用不到 Cursor、Claude Code 等编码工具；Manus 也是不温不火。再回到技术本身，例如长短期记忆、MCP、Skills 等，普通人基本上都接触不到。<font style={{ color: '#DF2A3F' }}>到头来，用的还是 Chat 对话，你一言，AI 一语；还是那个新建会话啥都忘记，啥都不会做的机器人。</font>

回过头来，普通人对于 AI 工具的诉求是什么？我觉得有以下几点：

+ <font style={{ color: '#DF2A3F' }}>能自定义机器人，并且记住之前做的事情 -> 长期记忆</font>
+ <font style={{ color: '#DF2A3F' }}>能真正做事情，例如管理邮件、分析股票、写文档等 -> MCP、Skills</font>

而这些 OpenClaw 做到了。

**从程序员的角度分析**

对于专业人士，具体指程序员，一款增强版 AI 编码工具（如 Claude Code、Cursor等）基本上就足够了，能用到长短期记忆、Skills、MCP等技术，能写代码、写文档、任务规划等。如果真有 AI 实现不了的功能，也能驱动 AI 写程序来完成特定的任务。但是在交互方式上，有以下问题：

+ 需要在特定电脑上，以特定方式进行对话，例如 CLI or IDE；
+ 一旦关了电脑，任务就停下来了。
+ 专注于编码任务，还是难以和日常生活中的其他任务打通，例如日程管理、邮件管理等。

<font style={{ color: '#DF2A3F' }}>总的来说，即使有了增强版 AI 编码工具，让人需要「程序员陪着 AI 做事情」，以及「难以打通其他工具」。而 OpenClaw 解决了这些问题，通过 24h 运行的 Server，24h 不间断的接收指令并完成任务。</font>

### 1.4 为什么 OpenClaw 能够火起来？
有以下一些原因：

+ 从使用人的角度，OpenClaw 解决了上一节提到的问题，让 AI 工具能够大范围用起来。
+ 从 AI 公司的角度，终于找到机会大范围卖 token 了。
+ 从 AI 技术发展的角度，通过技术的积累实现了工具的「进化」，成就了一款现象级产品。
+ “政治正确”：不管对于公司还是政府，「落后就得挨打」，必须推，也不得不推，自上而下的推。

### 1.5 OpenClaw 对程序员的影响是什么？
作为程序员，我持有的态度是**「不要恐慌」和「持续学习」**。

+ 不要恐慌：AI 对于程序员的冲击也不是一天两天了，也可预期的，未来只会有更多的冲击。没来一次技术更新都恐慌一次，实在是太累了，还不如静下心来，好好去了解学习背后的技术，不断的成长。
+ 持续学习：OpenClaw 的 Star 数量已经冲上了 github 第一，超越了 Linux 和 Redis，毫无疑问这是一款成功的应用。更何况还是开源的技术，那为什么不去学习它，将它转化为自己的知识，才能更好的应用它，并用在工作其他地方。

### 3.2 智能体循环 Agent Loop：ReAct 自我驱动的 Agent
> 官方把 Agent 执行的循环叫做 Agent Loop，但我更喜欢称之为 ReAct 范式的 Agent。
>

**(1) ReAct 模式的 Agent**

核心点在于实现了「ReAct 模式的 Agent」，实现了AI “规划-执行-反思”的循环。循环驱动类型有两种：

+ 等待用户对话后继续；
+ 执行 Tool 后自动继续。对于复杂任务，Tool 驱动的循环会更多，实现了类似持续工作的效果。

**其中，大量的循环是由 Tool 驱动自动完成，不需要用户介入，例如第 2.1.4 节。**

那么 ReAct 是如何驱动 Agent 呢？ReAct 类型的 Agent 主要是由 Prompt 驱动的，而不是 Workflow 等工程方法驱动。所以我们来看看他的提示词吧！具体见[「分析」System Prompt](https://www.yuque.com/xiaodai-tehze/lah8hg/bzhocz1ec68i4vg0)，其中的关键点在于：

+ 大量 tool 和 skills 的定义
+ ReAct 驱动：“先自己想办法，再开口询问。 先尝试搞清楚。读文件，看上下文，搜索一下。_然后_在卡住时再问。目标是带着答案回来，而不是带着问题。”

**(2) 底层具体实现：Pi Agent**

Pi Agent 的定位与 Claude Code 类似，是一款「增强版 AI 编码工具」。它的思想是尽可能简单，核心只使用了 <font style={{ color: 'rgb(25, 27, 31)' }}>read/write/edit/bash 四个 tool，追求「可预测性」Agent。这有一种「大道至简」的感觉，评测效果上也还不错。对比如下：</font>

### 3.5 自动任务执行：心跳与定时任务
> Ref：[https://docs.openclaw.ai/zh-CN/automation/cron-vs-heartbeat](https://docs.openclaw.ai/zh-CN/automation/cron-vs-heartbeat)

OpenClaw 中有两种自动任务执行的策略：

+ 心跳：每隔固定时间（默认30min）检查 HEARTBEAT.md 中指定的任务。
+ cron 定时任务：精准在某个时刻执行任务。

### 3.6 如何在 OpenClaw 执行的过程中插话？
> Ref：[https://docs.openclaw.ai/concepts/queue](https://docs.openclaw.ai/concepts/queue)
>

OpenClaw 使用 FIFO 队列来处理可能并发的消息。对于消息的处理有多种模式，具体如下：

假设：从模型开始生成，到输出纯文本回复用户为止的完整过程，中间可能包含多个 tool 调用。

```plain
run 开始
  → tool_use(A) → tool_result(A)
  → tool_use(B) → tool_result(B)
  → tool_use(C) → tool_result(C)
  → 纯文本回复用户
run 结束
```

假设：run 正在执行，已完成 A，B 正在跑，C D 还在队列里。用户此时发来新消息 X。

| 模式 | B（执行中） | C D（待执行） | X 何时注入 | 效果 |
| --- | --- | --- | --- | --- |
| `steer` | 跑完 | ❌ 取消 | B 完成后立即 | 改变方向，快速响应 |
| `followup` | 跑完 | ✅ 全跑完 | run 完全结束后 | 不打扰，排队等 |
| `collect` | 跑完 | ✅ 全跑完 | run 完全结束后，且合并多条消息 | 和 followup 一样，但多条消息合一 |


**关键差异总结：**

+ **要不要打断后续 tool？** steer / steer-backlog 会取消 C D；followup / collect 不会
+ **何时处理新消息？** steer 系列更快（mid-run），followup 系列更保守（end-run）
+ **collect vs followup？** 区别只在多条消息时：collect 合并成一条再处理，followup 逐条排队

## 五、总结与反思
### 5.1 「观点」未来需要更多面相 AI 的应用，为 AI 提供更好的交互环境
+ 我现在写文档会用语雀等在线文档，但无法让 AI 帮我润色，我期待语雀能够开发 API 允许 AI 像改本地文件一样，来帮我修改文档。
+ 不要和我说 Markdown + github 也能做到，或者用 obsidian。虽然能用 AI 工具本地编辑，但就是不方便。具体的：1）各类 markdown 工具语法支持的程度不一样；2）不支持分享或者多人协作等功能。
+ 类似的场景有很多。比如淘宝、支付宝、小红书等。

### 5.2 「思考」如何将 OpenClaw 应用到实际项目中？
对于个人或团队来说，OpenClaw 有助于编码等日常任务效率的提升。但怎么融入到一个真实的 toB 或 toC 的应用当中呢？

一些简单的思考：

+ 肯定不能完全套用。不可能为每个商家或者消费者提供一个独立的环境。
+ 如果共用一个环境，做复杂任务真的撑得住吗？
+ 做客服机器人是一个不错的选择，OpenClaw 本身也支持类似的功能。但似乎不需要本地环境执行指令。那就变成了24h响应，而不是24h做事情，那之前的 Agent 也能实现。
+ 定时任务和心跳机制也没什么意义。

**感觉 OpenClaw 大多数优势都被去除了，相当于「龙虾壳」被扒开了，只剩下「龙虾肉」了。剩下的有 ReAct 自主规划能力，Tools 和 Skills 的调用能力，长短期记忆管理能力，以及上下文压缩能力等。这些都是 Agent 开发能够用到的通用能力。对于应用开发，不需要小龙虾，而是要学习并应用其中 Agent 开发的思想。**
